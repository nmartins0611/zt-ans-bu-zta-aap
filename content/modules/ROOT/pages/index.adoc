== Purpose

This workshop shows a **Zero Trust Architecture** where **Red Hat Ansible Automation Platform (AAP)** is the main **Policy Enforcement Point (PEP)**: operational change (deploy, patch, network, incident response) goes through automation that consults **identity**, **policy**, **secrets**, and **infrastructure state**.

**Open Policy Agent (OPA)** acts as a **Policy Decision Point (PDP)**. **HashiCorp Vault** issues **short-lived** database credentials and **SSH certificates**. **IdM (FreeIPA)** provides directory identity and groups. **NetBox** is the **source of truth** for inventory. **SPIFFE/SPIRE** proves **workload identity** for sensitive jobs. **Splunk** or **Wazuh** plus **Event-Driven Ansible (EDA)** demonstrate **assume breach** response.

Lab DNS names use the `*.zta.lab` zone; replace with instructor-provided hosts where needed.

== Machines and components

|===
| Component | Role | Typical host / placement

| **AAP Controller**
| Automation orchestration (PEP); job templates, credentials, inventory
| `control.zta.lab` (dedicated VM)

| **EDA Controller**
| Event-driven automation; webhooks from SIEM → trigger AAP jobs
| Same controller VM as AAP in this lab (`control.zta.lab`)

| Identity, LDAP, Kerberos, CA, DNS
| `central.zta.lab`

| Policy decisions (deny-by-default Rego)
| `central.zta.lab` (e.g. port 8181)

| SSO / OIDC (workshop / future integration)
| `central.zta.lab` (container)

| **SPIRE Server**
| Trust root for SPIFFE workload identity
| `central.zta.lab`

| **SPIRE Agents**
| Workload attestation, SVID issuance
| `control`, DB host, Vault VM (per your deployment)

| **HashiCorp Vault**
| Secrets, dynamic DB creds, SSH CA signing
| `vault.zta.lab` (dedicated VM in reference design)

| **NetBox**
| CMDB / source of truth for devices and IPAM
| `netbox.zta.lab` (dedicated VM)

| **Gitea**
| Git server for workshop project / GitOps
| `gitea.zta.lab`

| **Splunk**
| Log aggregation, saved searches, alerts → EDA
| `central.zta.lab` (container; UI often `:8000`)

| `central.zta.lab` (container)

| RHEL container, app tier e.g. `10.20.0.10`, SSH e.g. `:2023`, HTTP `:8081`

== Network planes

Management plane (`192.168.1.0/24`)::
VMs (central, AAP, Vault, NetBox) attach here. Published ports on **central** reach containerized services (Splunk, Wazuh, app/DB SSH, etc.) from your workstation.

Data plane (Podman networks behind cEOS)::
* **Data tier** — e.g. `10.30.0.0/24` (`net2`): DB, ceos2 as gateway.  
* **App tier** — e.g. `10.20.0.0/24` (`net3`): app, ceos3 as gateway.  
**Switch ACLs** enforce **micro-segmentation** between tiers (lab exercises wire this to policy and Vault).
